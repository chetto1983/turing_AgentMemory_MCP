## Task 2 — real_document_benchmark.py N=3 frozen-question replay
(baseline/06-gate/real-document-benchmark-run{1,2,3}.json)

Method: run directly on the host against the published MCP endpoint
(http://127.0.0.1:8095/mcp/). The running `turing-agentmemory-mcp` compose
service already talks to the real granite/BGE sidecars by default (compose.yaml
env defaults EMBED_BASE_URL=http://agentmemory-embed:8080,
RERANK_BASE_URL=http://agentmemory-rerank:8080) — no override needed for this
leg; the script only drives MCP tool calls over HTTP, all embed/rerank work
happens server-side.

Command (run 3 times, N=1,2,3; PROVIDER_API_KEY set to a dummy value because
QuestionGenerator's constructor requires a non-empty key even though
`--frozen-questions` bypasses `generate()` entirely — no real LLM call is made):

  PROVIDER_API_KEY=unused-frozen-questions-bypass PYTHONPATH=src \
    .venv/Scripts/python.exe scripts/real_document_benchmark.py \
    --frozen-questions baseline/03-turingdb/frozen-questions.json \
    --root D:/tmp/baseline-corpus --mcp-url http://127.0.0.1:8095/mcp/ \
    --top-k 20 --search-concurrency 3 --chunk-bytes 524288 --poll-seconds 10 \
    --output baseline/06-gate/real-document-benchmark-run<N>.json

Each run uses a FRESH default `--scope` (the script derives
`scope = benchmark_id` when `--scope` is not passed, and `benchmark_id`
embeds a UTC timestamp), so each of the 3 runs ingests into its own new
ArcadeDB tenant database — no cross-run contamination, no reuse of any
document/tenant state pre-existing in the shared `turing-data`/ArcadeDB
volumes from earlier, unrelated container testing.

## GLiNER-scope decision (D-06)

Decision: ACCEPT the compose default (GLINER_ENABLED=1) rather than adding a
compose override to disable it for this capture. Rationale (matches the
plan's own documented low-risk-deviation option): `document_search` consumes
no entity channel in its ranking (RESEARCH.md Area 4 / Pitfall 4 — GLiNER
entity extraction only affects INGESTION-time processing and the separate
entity/fact retrieval channels used by memory_search, not document_search's
score), so the locked comparison metrics (mrr_at_20, recall_at_1, recall_at_20)
and the per-query SEARCH latency captured here are unaffected by GLiNER being
on. Only document ingestion wall-clock is heavier with GLiNER on (irrelevant
to the D-06 per-query retrieval-latency comparison against the baseline's
5.4s mean, which measures search latency, not ingest latency). Modifying
compose.yaml to add a GLINER_ENABLED=0 override was judged unnecessary
extra surface for a manual measurement plan whose declared file scope is the
baseline/06-gate/* artifacts only.
