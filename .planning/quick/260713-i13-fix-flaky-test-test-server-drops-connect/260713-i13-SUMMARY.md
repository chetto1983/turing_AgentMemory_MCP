# Phase quick-260713-i13 Plan 01: Fix flaky test_server_drops_connections_above_worker_cap Summary

Broadened one `except` clause (ConnectionAbortedError/ConnectionResetError to OSError) in `test_server_drops_connections_above_worker_cap` to tolerate the ENOTCONN socket-teardown race, with no product/source code changes.

## What Was Built

In `tests/test_gliner_provider_http.py::test_server_drops_connections_above_worker_cap`, the inner `except (ConnectionAbortedError, ConnectionResetError):` clause around the `raw_http_request(...)` call was replaced with `except OSError:`, with a comment explaining the non-obvious race: the server's `shutdown_request` (triggered by the worker-thread cap being saturated) can tear the client socket down before the client's own `client.shutdown(socket.SHUT_WR)` call completes, which raises a bare `OSError [Errno 107] ENOTCONN`. `ENOTCONN` is not a `ConnectionError` subclass, so it previously escaped the narrower except and failed the test on an otherwise-correct CI run.

The change is scoped to exactly this one except clause inside this one test function. The shared `raw_http_request` helper (used by two other tests that expect and must keep surfacing real socket errors) is untouched, and no `src/` module was modified — the server's `shutdown_request` behavior remains correct as-is.
