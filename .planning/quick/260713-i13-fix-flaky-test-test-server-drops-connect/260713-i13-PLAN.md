<objective>
Fix the flaky CI test `tests/test_gliner_provider_http.py::test_server_drops_connections_above_worker_cap`.

When the server correctly drops an over-cap connection via `shutdown_request`, the
client-side `raw_http_request` helper may raise `OSError [Errno 107] ENOTCONN` from
`client.shutdown(socket.SHUT_WR)` (test_gliner_provider_http.py:68) if the server tore
the socket down first. `ENOTCONN` is a plain `OSError`, NOT a `ConnectionError` subclass,
so the test's `except (ConnectionAbortedError, ConnectionResetError)` at line ~314 does
not catch it — the exception escapes and CI goes red on an otherwise-correct run.

Purpose: The failure is a FLAKY TEST, not a product bug. Root cause is already
diagnosed (see rationale below). The server's `shutdown_request` on semaphore
saturation is correct and MUST NOT change.
Output: A one-line `except` broadening (test file only) plus a short "why" comment.

Rationale (CLAUDE.md "NEVER MODIFY TESTS TO MAKE THEM PASS" justification): the test IS
genuinely broken — its exception handling is incomplete for a legitimate race outcome
(server-side socket teardown before the client's half-close). The product code is
correct. This justification MUST be captured in the commit body.
</objective>
