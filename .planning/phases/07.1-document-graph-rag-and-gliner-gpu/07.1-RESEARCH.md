## Summary

This phase has two very different kinds of work, and the research effort split accordingly.

**GLiNER GPU/batching (PERF-04, FIX-08)** turned out to rest on a wrong mental model in
CONTEXT.md's own framing. `fast_gliner` is not a wrapper around the Python `onnxruntime` /
`onnxruntime-gpu` packages ŌĆö it is a Rust binding (PyO3, compiled `.so`/`.pyd`) around
`gline-rs`, which itself wraps the Rust `ort` crate (a Rust binding to ONNX Runtime).
Downloading and unzipping the exact pinned `fast_gliner==0.2.1` wheel from PyPI confirms: (1) the
plain `pip install fast_gliner` the current Dockerfile runs installs a **prebuilt CPU-only**
wheel ŌĆö CUDA requires `pip install --no-binary=:all: fast_gliner[cuda]`, a from-source build
needing a Rust toolchain AND the CUDA toolkit in the build stage, not just a CUDA-capable base
image; (2) `FastGLiNER2.predict_entities` explicitly raises if given more than one text at a time
(confirming the `del batch_size` comment is accurate for 0.2.1 and there is no batched
`predict_entities` path at all ŌĆö a real batching win, if wanted, is out of `fast_gliner`'s public
API entirely); (3) there is no Python-level introspection API to prove which execution provider
actually bound at runtime ŌĆö the silent-CPU-fallback behavior CONTEXT.md worried about is
independently confirmed by the upstream `ort`/`gline-rs` documentation as the *designed* behavior,
with no logging on failure at any layer. The binding-proof test the phase requires must therefore
be a black-box (timing or `nvidia-smi` utilization) test, not a white-box provider-query test.

**The ArcadeDB/entity-resolution/eval side (GRAPH-01..04)** is comparatively low-risk: every
query surface this phase needs (chunk-entity `MENTIONS` writes, multi-hop entity-bridge
traversal, the `RetrievalCandidate`ŌåÆ`fuse_rankings`ŌåÆ`diversify_fused` shape) already has a
directly-precedented, live-validated implementation to copy from ŌĆö either production code
(`store_evidence.py`, `store_retrieval_queries.py`) or spike 003's `run.py`, which is a
throwaway prototype but was run against real ArcadeDB/GLiNER/BGE and is the single best template
for the eval harness's three-arm driver. One real correction to CONTEXT.md surfaced during this
research: `memory_rebuild_vector_projection`/`memory_rebuild_communities` re-embed/re-cluster
**existing** Entity/Fact rows ŌĆö they do not re-derive entity identity from source text, so they
alone cannot execute D-02's re-keying on already-ingested data (see "Unknown 8a").

**Primary recommendation:** treat PERF-04 as a Dockerfile-plus-Rust-toolchain problem (not a
`onnxruntime-gpu` pip-swap problem), design the binding-proof test as a measured throughput/GPU-
utilization delta (not a provider-string assertion), and build the document graph substrate by
directly reusing `store_evidence.py`'s traversal-statement conventions plus spike 003's
validated write/read patterns ŌĆö no new query mechanism needs to be invented.

## Resolved Unknowns

### Unknown 1 ŌĆö `fast_gliner==0.2.1` batching API (D-09, HIGHEST VALUE)

**Evidence:** Downloaded `fast_gliner-0.2.1-cp312-cp312-manylinux_2_28_x86_64.whl` directly from
PyPI (`files.pythonhosted.org`, sha256 `3496cb6b62ded91e504240706fd8056cccc7576a0607ecbb60430df589a6659f`)
and unzipped it. `fast_gliner/__init__.py` (344 lines, quoted verbatim in the extraction) is the
canonical Python API surface `gliner_provider.py:89` calls into.

```python
# fast_gliner/__init__.py:245-261 (exact quote from the unzipped wheel)
class FastGLiNER2(_FastGLiNERBase):
    _backend = PyFastGliNER2

    def predict_entities(
        self, input_text: Union[str, List[str]], labels: List[str]
    ) -> Union[List[dict], List[List[dict]]]:
        """
        Run NER inference using GLiNER2.

        Note
        ----
        GLiNER2 currently does **not support batched inference**.
        """
        if isinstance(input_text, list) and len(input_text) > 1:
            raise ValueError(
                "GLiNER2 currently does not support batched inference. Please pass a single input string."
            )
        return super().predict_entities(input_text, labels)
```

**Finding: `[VERIFIED: fast_gliner==0.2.1 wheel]` ŌĆö the `del batch_size  # FastGLiNER2 currently
accepts one text per inference.` comment at `gliner_provider_extraction.py:61,96` is exactly
accurate for the pinned version.** There is no way to submit a batch of texts to
`FastGLiNER2.predict_entities` at all ŌĆö the library itself raises `ValueError` if you try.
`GLINER_BATCH_SIZE` cannot become "real" by any change on this repo's side; batching a real
speedup requires either (a) upstream `fast_gliner`/`gline-rs` adding a batched entry point in a
future release (not available in 0.2.1), or (b) parallelizing multiple single-text
`predict_entities` calls across threads/processes at the Python level in
`FastGLiNER2Adapter.batch_extract_entities` (`gliner_provider_extraction.py:63-87`) ŌĆö e.g. a
`ThreadPoolExecutor` fan-out per HTTP request, since the underlying Rust call releases the GIL
during inference (standard for PyO3-wrapped native compute). This is a **concurrency**
improvement, not a **model-level batching** improvement, and is the only real lever D-09 has for
"a real batched inference path" against this exact library version.

**Secondary finding, also load-bearing:** `FastGLiNER2.extract_relations` (used for
`extract_relations` in `_extract_relations_common`, base class) has NO such single-item
restriction ŌĆö only `predict_entities` is restricted for GLiNER2. This is irrelevant to this
phase's entity-extraction path (`gliner_provider.py`'s HTTP `/extract` endpoint only calls
`predict_entities`, not `extract_relations`), but is worth recording so a future phase doesn't
assume the restriction is universal.

**Recommendation for the plan:** design D-09's "real batching path" as concurrent single-item
inference calls (thread pool sized to the sidecar's CPU/GPU capacity) inside
`FastGLiNER2Adapter.batch_extract_entities`, NOT as passing a list to `predict_entities`. Retire
or repurpose `GLINER_BATCH_SIZE` to mean "concurrency fan-out width" rather than "model batch
size" ŌĆö and document that rename/repurposing explicitly, since the env var name is misleading
against the true mechanism.

### Unknown 2 ŌĆö CUDA execution-provider binding proof (D-09)

**Evidence chain (all independently corroborating):**

1. `[VERIFIED: fast_gliner wheel SBOM]` ŌĆö `fast_gliner-0.2.1.dist-info/sboms/fast_gliner.cyclonedx.json`
   lists Rust dependencies `gline-rs 0.9.5-SNAPSHOT`, `ort 2.0.0-rc.9`, `ort-sys 2.0.0-rc.9`. This
   is a Rust `ort`-crate binding (Microsoft ONNX Runtime via Rust FFI), not the Python
   `onnxruntime`/`onnxruntime-gpu` package ŌĆö `requires_dist` for the PyPI package lists ONLY
   `huggingface-hub<1.0,>=0.16.4`; there is no Python-level ONNX Runtime dependency at all.
2. `[CITED: raw.githubusercontent.com/talmago/fast_gliner/main/README.md]` ŌĆö three install paths:
   `pip install fast_gliner` (prebuilt, CPU), `pip install --no-binary=:all: fast_gliner` (source,
   CPU), `pip install --no-binary=:all: fast_gliner[cuda]` (source, CUDA). The Dockerfile's
   current plain `pip install "fast_gliner==0.2.1"` is definitively the first, CPU-only path.
3. `[CITED: WebSearch summary of gline-rs/ort crate documentation]` ŌĆö "you must activate the
   appropriate [Cargo] features, otherwise the example will silently fall-back to CPU" and "for
   prebuilt Microsoft binaries, you can enable the CUDA... execution providers... via the `cuda`
   ...Cargo feature[s]... If you haven't enabled the appropriate Cargo features, the library will
   silently fall back to CPU execution." This is a **compile-time** (Cargo feature), not merely a
   runtime, gate ŌĆö confirming and sharpening CONTEXT.md's "ONNX Runtime silently falls back"
   framing: even a CUDA-capable base image cannot fix this unless the wheel/build was compiled
   with `--features=cuda`.
4. `[CITED: same source]` ŌĆö the `ort` crate exposes `ExecutionProvider::cuda().is_available()`
   and a builder `.error_on_failure()` at the **Rust** level. Neither is reachable from Python:
   `fast_gliner/__init__.py`'s public surface (`FastGLiNER`, `FastGLiNER2`, `predict_entities`,
   `extract_relations`, `classify`, `create_schema`, `extract`, `extract_json`) exposes nothing
   that queries the bound execution provider. `gline-rs`'s own documentation confirms "no logging
   mechanism" on registration failure at any layer (Rust, `gline-rs`, or the Python binding).

**Conclusion (open risk, explicitly flagged):** there is **no white-box way** to assert "CUDA is
actually bound" from Python via `fast_gliner`'s public API in version 0.2.1. A test that queries
`InferenceSession.get_providers()` (the standard Python-`onnxruntime` idiom CONTEXT.md's phrasing
implicitly assumed) **does not apply here** ŌĆö that class/method does not exist in this binding's
call path at all.

**Recommendation ŌĆö the binding-proof test must be black-box:**
- **Primary signal:** GPU memory/utilization delta. Before/after a burst of `predict_entities`
  calls (e.g. 50 chunks from spike 003's 167-chunk corpus), sample
  `nvidia-smi --query-gpu=utilization.gpu,memory.used --format=csv,noheader` (already proven
  available inside these containers ŌĆö `agentmemory-embed`'s healthcheck at `compose.yaml:351`
  runs `nvidia-smi` successfully). If GPU utilization/memory does not move measurably while CPU
  utilization does not spike commensurately with the corpus size, CUDA is not bound. This is the
  same category of test the repo already runs for the embed/rerank sidecars via their healthchecks.
- **Secondary signal:** wall-clock throughput. `fast_gliner`'s own README claims "~4├Ś faster CPU
  inference than the PyTorch implementation" as a baseline; a CUDA-bound run over spike 003's
  167-chunk / 2215-chunk corpus should show an order-of-magnitude speedup over the CPU baseline
  (spike 003: >27 min for 2215 chunks on CPU). A binding-proof test asserting "GPU run completes
  in under N seconds where N is derived from a measured CPU baseline, with a large safety margin"
  is a legitimate, CI-appropriate proxy ŌĆö but must be paired with the GPU-utilization signal
  (a batching/concurrency fix from Unknown 1 could ALSO speed up CPU inference, so throughput
  alone is not conclusive; use both signals together).

### Unknown 3 ŌĆö ArcadeDB chunk-traversal query shape (GRAPH-02)

**Two live-validated forms exist in this codebase; use the simpler one for the new topology.**

1. **Production object-notation `MATCH`** (`store_retrieval_queries.py:121-158`,

   "WHERE id = :eid AND user_identifier = :u) WHERE user_identifier = :u LIMIT 60"

   the memory graph's multi-type chain. **Recommendation: use this `expand(in()/out())` form for

### Anti-Patterns to Avoid
- **Do not** pass a list to `FastGLiNER2.predict_entities` expecting batching ŌĆö it raises
  `ValueError` by design in 0.2.1 (Unknown 1).
- **Do not** treat `execution_provider="cuda"` accepting without exception as proof of GPU
  binding ŌĆö silent CPU fallback is the *documented default* behavior of the underlying `ort`
  crate (Unknown 2).
- **Do not** run `process_many`/entity extraction on whole-document text before chunking ŌĆö this
  is the exact, already-diagnosed cause of the HTTP 400 (Unknown 1/D-10 interaction).
- **Do not** copy `diversify_fused(max_per_source=1)` verbatim into the document-search fusion
  port (Unknown 6).
- **Do not** claim `memory_rebuild_vector_projection`/`memory_rebuild_communities` re-key
  existing entities ŌĆö they re-embed/re-cluster what's already there (Unknown 8a).
- **Do not** use the Cypher-arrow `MATCH (c:Chunk)-[:MENTIONS]->...` form anywhere ŌĆö confirmed
  dead on ArcadeDB 26.7.1 by spike 001, independently by Phase 4 D-05.

## Common Pitfalls

### Pitfall 1: Assuming `fast_gliner` is a Python-`onnxruntime` wrapper
**What goes wrong:** planning a Dockerfile fix as "swap `onnxruntime` for `onnxruntime-gpu`" or
writing a binding-proof test against `onnxruntime.InferenceSession.get_providers()`.
**Why it happens:** the class/parameter names (`execution_provider`, "ONNX Runtime") strongly
suggest the standard Python onnxruntime idiom.
**How to avoid:** re-read Unknown 1/2 above; the actual mechanism is a Rust `ort`-crate binding
with Cargo compile-time features, requiring a from-source build with a Rust+CUDA toolchain.
**Warning signs:** any task description mentioning `onnxruntime-gpu` as a pip package to add.

### Pitfall 2: Building the document-side entity extraction on the whole document text
**What goes wrong:** the exact HTTP 400 this phase is meant to fix, reproduced.
**Why it happens:** `_process_text_for_storage` is called on `text` before `_chunk_document_text`
in the CURRENT code ŌĆö copy-pasting the current call order into a "fixed" version without moving
the extraction step is an easy mistake.
**How to avoid:** extraction must run on the CHUNK list, in Ōēż256-item sub-batches, AFTER chunking.
**Warning signs:** any diff that still calls `_process_text_for_storage(text, ...)` with the raw,
un-chunked `text` variable.

### Pitfall 3: Copying `diversify_fused(max_per_source=1)` into document search
**What goes wrong:** documents that should return 3-5 relevant chunks return only 1 per document,
regardless of overall relevance ŌĆö silently degrades quality with no obvious error (Unknown 6).
**How to avoid:** explicitly pass `max_per_source > 1` when calling `diversify_fused` for the
document-search fusion port; add a regression test asserting a single document CAN contribute
more than one chunk to a result set.

### Pitfall 4: Trusting `memory_rebuild_vector_projection`/`memory_rebuild_communities` to fix
existing fragmented entities
**What goes wrong:** running these tools after the D-02 code change, observing no improvement in
cross-doc bridge counts, and concluding the fix "didn't work" ŌĆö when in fact the fix landed
correctly for new data but old data was never touched (Unknown 8a).
**How to avoid:** for any tenant/corpus this phase measures against, re-ingest from source after
the code change, per the fresh-start recommendation above.

### Pitfall 5: Conflating the eval's "unanswerable from one chunk" filter with its GO/NO-GO metric
**What goes wrong:** using the same function for question REJECTION (single-passage
answerability check, D-07) and for the actual bridging-set recall@k measurement (D-08) ŌĆö these
need different logic (single-rank vs. set-recall) even though both build on the same
substring-matching primitives (Unknown 5).

## Open Questions

1. **Does the `cuda`-feature `fast_gliner[cuda]` build, once compiled, need ANY additional
   runtime configuration (e.g. an `ORT_DYLIB_PATH` env var, a bundled `libonnxruntime_providers_cuda.so`)
   beyond what a standard CUDA-runtime base image + `execution_provider="cuda"` provides?**
   - What we know: the Cargo `cuda` feature gates COMPILE-TIME inclusion of the CUDA execution
     provider in the linked ONNX Runtime binary (per the `ort`/`gline-rs` documentation fetched).
   - What's unclear: whether the `ort` crate's default build mode statically links a
     Microsoft-provided prebuilt ONNX Runtime binary (in which case no separate runtime dylib is
     needed) or dynamically loads one (in which case the CUDA runtime shared libraries must ALSO
     be present in the final image, separate from the CUDA toolkit used at build time).
   - Recommendation: the cheapest resolution is empirical ŌĆö build the image, run the
     binding-proof test (Unknown 2's recommendation), and let the black-box GPU-utilization signal
     answer this directly rather than trying to fully resolve the Rust build system's linking mode
     from documentation alone.

2. **Exact `max_per_source` value and `entity_boost`/`fused_graph` fusion weight values for the
   document-side `fusion_weights` dict extension.**
   - What we know: `store_core.py:195-205`'s `fusion_weights` dict is instance-level and shared
     across memory and document fused search; new channel-name keys must be added for whatever
     document-side channel names the D-14 port introduces (`fuse_rankings` hard-requires a weight
     per channel used).
   - What's unclear: whether document channels should reuse the SAME weight keys as memory search
     (e.g. `"graph": 0.5` shared) or get distinct keys (e.g. `"doc_graph"`) ŌĆö reusing shared keys
     risks unintentionally coupling memory-search tuning to document-search tuning; distinct keys
     avoid that but need their own defaults chosen.
   - Recommendation: use distinct, `doc_`-prefixed channel keys (`doc_dense`, `doc_lexical`,
     `doc_graph`) to keep memory-search and document-search tunable independently, mirroring the
     mem0-corroborated `ENTITY_BOOST_WEIGHT = 0.5` sub-full-weight guidance CONTEXT.md D-13 already
     cites for the graph channel generally.
