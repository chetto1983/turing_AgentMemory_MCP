<objective>
Make `GLINER_BATCH_SIZE` mean something real: a concurrency fan-out width over single-text
inference calls — and document, honestly, that model-level batching is impossible against the
pinned library.

Purpose: **`GLINER_BATCH_SIZE` is inert today.** `gliner_provider_extraction.py:61` reads
`del batch_size  # FastGLiNER2 currently accepts one text per inference.` — compose sets `1`, the
code default is `8`, and **both are decorative**. There is no batching on any device, which is a
major reason spike 003 saw 2215 chunks take >27 min. The research verified against the unzipped
pinned wheel that `FastGLiNER2.predict_entities` **raises `ValueError`** on a list of >1 text, by
design (`fast_gliner/__init__.py:245-261`). So D-09's "real batching path" **cannot** be model
batching. The only real lever is parallelizing single-text calls: the underlying Rust call releases
the GIL, so a thread pool is a genuine throughput win.
Output: a concurrent `batch_extract_entities` that preserves order and count, and an honest
account of what `GLINER_BATCH_SIZE` now means.
</objective>

<feature>
  <name>Concurrency fan-out for single-text GLiNER inference</name>
  <files>src/turing_agentmemory_mcp/gliner_provider_extraction.py, tests/test_gliner_provider.py</files>
  <behavior>
    `FastGLiNER2Adapter.batch_extract_entities(texts, labels, batch_size=N, threshold, include_confidence, include_spans)`:
    - Returns a list with `len(results) == len(texts)`, and `results[i]` corresponds to `texts[i]` — INPUT ORDER, regardless of which thread finishes first. Assert with a fake model whose per-text latency is deliberately inverted (text 0 slowest, text N-1 fastest) so a naive as-completed implementation fails.
    - Calls `model.predict_entities` exactly once per text, always with a `str` — never a `list`. Assert with a fake model that raises if handed a list (mirroring the real 0.2.1 contract).
    - `batch_size` bounds concurrency: with `batch_size=2` and 8 texts, no more than 2 calls are ever in flight simultaneously (assert with a fake model tracking a concurrent-call high-water mark).
    - `batch_size=1` is sequential and still correct (order + count preserved).
    - `texts=[]` returns `[]` and never constructs a pool.
    - Every existing per-entity filter still applies identically after the fan-out: sub-threshold scores dropped, non-finite/bool scores dropped, non-dict entries skipped, `_normalize_entity_offsets` applied, `score`/`start`/`end` popped per `include_confidence`/`include_spans`.
    - If `predict_entities` raises for ONE text, the error surfaces (it is not swallowed into an empty result) and no partial/misaligned list is returned.
    - A non-list return from `predict_entities` still raises `ValueError("FastGLiNER2 returned a non-list result")`.
  </behavior>
  <implementation>
    Extract the current per-text loop body into a helper, submit it to a bounded
    `ThreadPoolExecutor`, and collect by input index. See tasks.
  </implementation>
</feature>
