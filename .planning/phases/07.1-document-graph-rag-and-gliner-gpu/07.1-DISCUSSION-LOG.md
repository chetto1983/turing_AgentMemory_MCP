contain multi-hop questions?) with **efficacy** (does the channel retrieve better on them?). A
MuSiQue-style eval is multi-hop *by construction*, so it measures efficacy and says nothing about
demand.

## Threshold Number

**Claude raised the `entity_boost` control arm before asking:** mem0's *replacement* for its
deleted graph — a name-keyed entity inverted index with a hub-damped single-hop boost —
apparently beat the graph it replaced. Multi-hop must beat that cheap baseline, not just beat
no-graph.

| Option | Description | Selected |
|--------|-------------|----------|
| Strict, vs best cheap baseline | 3 arms; GO requires ≥+0.10 mean recall@10 over the BEST cheap arm, ≥⅓ questions improved with zero regressions, no >0.03 regression on the frozen yardstick, latency ≤+20% | ✓ |
| Moderate, vs `fused_base` only | 2 arms; ≥+0.05 mean recall@10 over base, no regression | |
| Set the number from baseline first | Run cheap arms, set threshold from observed baselines, then run `fused_graph` | |

**User's choice:** Strict, vs best cheap baseline
**Notes:** The strongest decision in the session. The eval is stacked *in the channel's favour*
(multi-hop by construction), the literature ceiling is small (HippoRAG 2: HotpotQA +0.2 **with**
an LLM write path), and the named failure mode is "build it, measure +0.2, call it a win because
it's already built."
