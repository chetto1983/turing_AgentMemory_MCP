<objective>
Ship a tenant-scoped re-key rebuild that re-derives entity identity from canonical in-database
text, so D-02's name-only key actually applies to already-stored data (D-19).

Purpose: **`rebuild_vector_projection` re-embeds and `rebuild_communities` re-clusters rows that
ALREADY EXIST — neither re-runs extraction nor re-derives an entity's `stable_id` from source
text** (verified by reading `store_rebuild.py:51-90` and `:239-241` in full). So plan 03's re-key
would apply **only to newly-ingested data**, leaving every already-stored entity fragmented under
the old key — and D-02's measured 3x-cross-doc-bridge benefit would never materialize. D-04's
stated *mechanism* is factually wrong; its *intent* stands and is delivered here.

**This is a rebuild, not a re-ingest.** CLAUDE.md already promises derived projections are
"tenant-scoped and rebuildable from canonical data"; `Memory.content` / `Chunk.text` **are**
canonical and still in the database. The gap is that the rebuild tool the architecture claims to
have does not re-derive identity. This closes that gap (fix-on-touch).

**Do NOT plan or execute a "fresh re-ingest" of the `dvdmarchetto@gmail.com` tenant.** Research's
re-ingest recommendation rests on a false premise — it assumed no production tenant data, which is
true for *customer* data but **false for this repo's own dogfooding tenant**. Those memories were
written via `memory_add_fact` / `memory_store_message`: **there are no source files; the store IS
the source.** "Drop and re-ingest" would destroy them. Re-ingest is viable only for the E2E gate's
synthetic tenants and the eval corpus, which do have source files.

**D-03's counterweight is a PRECONDITION of this plan, not a follow-up.** Re-keying deliberately
creates hubs — collapsing `arcadedb`'s three fragments makes one node linked to 3x more memories —
and D-03 says the ER fix "regresses precision without this counterweight; it cannot ship without
it." D-19 restates it: *the rebuild and the damping land together or not at all.* Plan 17 (wave 2)
wires `hub_damping_factor` into the memory graph channel in `store_evidence.py`, and this plan
`depends_on` it, so the counterweight is active before this rebuild manufactures a single hub. If
plan 17's SUMMARY does not record the damping as ACTIVE on the memory path, **stop** — the
precondition is unmet and this rebuild must not run.
Output: a re-key rebuild entry point, its MCP tool, and tests pinning its four hard constraints.
</objective>

  <behavior>
    `rebuild_entity_identity(user_identifier=...)`:
    - `user_identifier=""` (or whitespace-only) raises immediately, before ANY read or write — fail closed (invariant #1)
    - Given a tenant whose stored entities were keyed `(entity_type, canonical_name)` — e.g. three rows for "arcadedb" as product/library/framework — a run collapses them to ONE entity at `stable_id("ent", user_identifier, "arcadedb")`, with `type_observations` == `{"product": N1, "library": N2, "framework": N3}` reflecting the observed counts
    - Every edge that pointed at any of the three old ids (`MENTIONS`, `SUBJECT_OF`, `OBJECT_OF`, `SUPPORTED_BY`) points at the surviving id afterwards — ZERO orphaned edges, asserted by counting edges whose endpoint id no longer exists (must be 0)
    - Canonical rows are untouched: `Memory` / `Chunk` / `Document` row counts and their `content`/`text` values are byte-identical before and after
    - Idempotent: running twice yields the same entity count, the same ids, the same `type_observations`, and the same edge count as running once
    - A second tenant's entities are completely unaffected by a run scoped to the first (row-level diff == empty for tenant B)
    - Returns a result dict reporting per-kind counts (entities before/after, facts, mentions rewritten) — mirroring `rebuild_vector_projection`'s result shape
    - The whole rewrite for one tenant commits through ONE `run_in_transaction` cycle (invariant #5)
    - An MVCC 503 / `ConcurrentModificationException` redoes the WHOLE begin/body/commit cycle, bounded by `ARCADEDB_COMMIT_RETRIES` — never a blind re-commit (invariant #6)
  </behavior>
  <implementation>
    A new mixin reading canonical text, re-running extraction, and rewriting derived rows via
    UPSERT + edge re-pointing. See tasks.
  </implementation>
</feature>
