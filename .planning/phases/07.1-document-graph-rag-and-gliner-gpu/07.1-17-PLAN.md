  <behavior>
    Query-side seed derivation (`_query_graph_evidence`):
    - A query mentioning one surface under a drifting GLiNER label yields a seed id EQUAL to
      `stable_id("ent", user_identifier, canonicalize_entity_name(text))` — the same id plan 03's
      write path stores. Today the two disagree and the channel would seed nothing.
    - Two extracted mentions of the SAME surface under DIFFERENT labels collapse to ONE seed whose
      score is the max of the two (the existing `max(...)` merge already does this once the ids agree
      — do not rewrite it).
    - `user_identifier` remains the first key part (invariant #1, unchanged by D-02).

    Hub damping (`_expand_entity_evidence`):
    - An entity linked to 1 memory produces candidates whose `raw_score` is unchanged
      (factor exactly 1.0).
    - An entity linked to 100 memories produces candidates whose `raw_score` is ~0.09x the undamped
      value (assert the score ratio to 2dp, not merely that it is smaller).
    - Damping COMPOSES with hop decay: a hop-2 candidate for a 100-memory entity scores
      `seed_score * confidence / 2 * ~0.09` — both mechanisms apply, neither replaces the other.
    - An entity with NO count row damps by 1.0 (never dropped, never zero-scored).
    - The counts read is issued ONCE per expansion, not once per traversal and not once per row.
    - The counts read binds `user_identifier`; a foreign-tenant fixture yields zero counts and hence
      zero candidates.
  </behavior>
  <implementation>
    A bound-param count builder plus a damping multiplication at the single shared entity-expansion
    scoring site. No new ranking mechanism; `hub_damping_factor` is imported from plan 04, never
    re-derived. See tasks.
  </implementation>
</feature>
