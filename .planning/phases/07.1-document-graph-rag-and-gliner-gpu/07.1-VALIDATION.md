## Manual-Only Verifications

| Behavior | Requirement | Why Manual | Test Instructions |
|----------|-------------|------------|-------------------|
| **CUDA binding proof** — GLiNER actually runs on GPU, not silently on CPU | PERF-04 (D-09) | **No white-box API exists.** `fast_gliner==0.2.1` is a Rust/PyO3 binding around `gline-rs`/`ort`, not the Python `onnxruntime` package — there is no `InferenceSession.get_providers()` in the call path, and silent CPU fallback is the *documented default* of the underlying `ort` crate (research Unknown 2). Requires real GPU + Docker. | **Use BOTH signals together — neither alone is conclusive.** (1) GPU delta: sample `nvidia-smi --query-gpu=utilization.gpu,memory.used --format=csv,noheader` before/after a burst of ≥50 chunk extractions; GPU utilization/memory must move measurably. (2) Throughput floor: a CUDA-bound run over spike 003's 167-chunk set must beat a *measured* CPU baseline by a large margin. Throughput alone is insufficient — the concurrency fan-out (PERF-04) also speeds up CPU. **Never** treat wheel-install success or `execution_provider="cuda"` being accepted without exception as proof. |
| **GPU contention measurement** — effect of a third GPU model on embed/rerank latency | PERF-04 (D-11) | Requires the full GPU compose stack; contention is only observable under real concurrent load. | Enable GLiNER on GPU with embed+rerank running; measure embed/rerank latency and the E2E score gate against the documented 3345 ms mean vs the 5.4 s bar. **Measure before tuning anything** (D-11). |
| **D-17 mini migration-correctness gate** | GRAPH-01, GRAPH-04 | Requires live GPU stack + real providers + the committed corpus; mirrors Phase 6's gate. | After the re-key rebuild (D-19) lands: re-run `scripts/e2e_score.py` — must read **19/19 at 10.0** (`VALIDATED_10_10`, `score >= 9.8 and check_count == 19`). Re-run the frozen-question benchmark via `scripts/gate_diff.py` (same CLI Phase 6 used) to confirm the document path is untouched with the flag OFF. Record as a `GATE.md`-shaped artifact. |
| **Multi-hop eval + GO/NO-GO verdict** | GRAPH-03 | The decision artifact itself; requires real providers, the corpus, and **user review** of surviving questions (D-07: "Claude composes, user verifies"). | Compose candidate two-hop chains; mechanically cut any question the current dense+lexical+rerank stack answers from a single chunk; user reviews survivors for plausibility. Run all three arms (`fused_base` / `entity_boost` / `fused_graph`). Apply D-15's four GO conditions — **all four required**. |
