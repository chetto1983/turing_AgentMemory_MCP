<objective>
Prove — black-box — that GLiNER is actually running on the GPU, measure the throughput change
honestly, and measure GPU contention against embed/rerank before tuning anything.

Purpose: this plan exists because **you can "enable GPU", measure nothing, and get no error.**
ONNX Runtime's silent CPU fallback is the *documented default* of the underlying `ort` crate, with
no logging on registration failure at any layer. There is **no white-box way** to assert CUDA is
bound: `InferenceSession.get_providers()` — the standard Python-`onnxruntime` idiom CONTEXT.md's
phrasing implicitly assumed — **does not exist in this call path at all**. `fast_gliner==0.2.1`'s
public surface exposes nothing that queries the bound execution provider. The binding proof must
therefore be black-box, and it must use two signals, because plan 06's concurrency fan-out also
speeds up CPU — so throughput alone would be fooled by our own other change.
Output: a two-signal binding-proof test, a probe script, and a measurement record.
</objective>

    Create `tests/test_gliner_gpu_binding.py` asserting BOTH signals TOGETHER:
      1. **GPU delta:** GPU utilization/memory must move measurably across the burst.
      2. **Throughput floor:** the run must beat a **measured** CPU baseline by a large margin.

    Neither alone is sufficient and the test must not pass on one. The throughput signal alone is
    specifically insufficient because plan 06's concurrency fan-out also speeds up CPU — a
    CPU-bound run with fan-out could clear a naive floor. The GPU signal alone is insufficient
    because another container shares the device (see Task 2).

    `GLINER_DEVICE=cpu` and record the number. A floor derived from a guessed baseline proves
    nothing. Commit the measured baseline as data the test reads, mirroring how
    `baseline/03-turingdb/` freezes a measured bar.

    On a GPU-less runner the test must either run the same black-box check against CPU and assert a
    *documented* CPU-tier floor — mirroring the existing `HashingEmbedder`-stub pattern the E2E gate
    already uses, and the measured stub floor CI asserts per STATE.md's 01-09 entry — or fail
    loudly. Never silent green.

    **Never** treat wheel-install success, or `execution_provider="cuda"` being accepted without an
    exception, as proof of binding. Both succeed silently on the CPU-only prebuilt wheel. Do not add
    an assertion of that shape even as a convenience check — it would read as evidence and is not.
