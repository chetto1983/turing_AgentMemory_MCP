  <behavior>
    `hub_damping_factor(n)` where n = the number of chunks (or memories) an entity is linked to:
    - n=1  -> exactly 1.0 (a singleton entity is not penalised at all)
    - n=5  -> approximately 0.98 (CONTEXT.md D-03's stated worked value; assert to 2 decimal places)
    - n=100 -> approximately 0.09 (CONTEXT.md D-03's stated worked value; assert to 2 decimal places)
    - Monotonically non-increasing in n across n = 1..200
    - Always in the half-open interval (0.0, 1.0] — never 0, never negative, never > 1
    - n=0 and negative n are defensive inputs: return 1.0 rather than raising (a hubless/unlinked
      seed must not be silently dropped from the channel)
    `diversify_fused` (existing function, characterization only — no code change):
    - With `max_per_source=1` and 3 fused candidates whose `source_memory_id` is the SAME document
      id, exactly 1 survives (this is the trap: correct for memories, wrong for documents)
    - With `max_per_source=3` and the same input, all 3 survive
    - The cap groups by `source_memory_id`, falling back to `candidate_id` when it is empty
  </behavior>
  <implementation>
    A pure arithmetic function plus tests over the existing `diversify_fused`. No fusion behavior
    changes in this plan — the damping is wired into the graph channel in plan 13.
  </implementation>
</feature>
