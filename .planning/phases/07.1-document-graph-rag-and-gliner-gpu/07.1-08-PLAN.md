<objective>
Fix the whole-document → HTTP 400 ingest gap (D-10) by moving entity extraction to run per-chunk
after chunking, sub-batching to `MAX_TEXTS=256`, and giving each chunk its own extraction metadata.

Purpose: `ingest_document_text` calls `_process_text_for_storage(text, metadata)` on the **whole
document text** at `store_documents.py:123`, while chunking happens at `:126` — AFTER. That
ordering, plus `MAX_TEXT_CHARS = 16_384` (`gliner_provider_extraction.py:22`), IS the HTTP 400: any
document past a few pages blows the bound and `_validate_string_list` raises `RequestFailure()`
(default `BAD_REQUEST`). Spike 003 crashed on this; the Phase-3 baseline only dodged it by running
GLiNER off. Per-chunk extraction is **on the critical path, not adjacent to it** — it is exactly
what plan 10's `(:Chunk)-[:MENTIONS]->(:Entity)` substrate needs.
Output: documents of any size ingest with per-chunk entity metadata.
</objective>

|----------|-------------|
| untrusted document text -> GLiNER sidecar `/extract` | Document content of arbitrary size and shape crosses into an external HTTP service with hard payload bounds. |
| GLiNER response -> Chunk.metadata_json | Extraction output is persisted per chunk and later drives graph edges (plan 10). |

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |

| T-07.1-03 | Denial of Service | GLiNER sidecar payload-size DoS — an oversized or over-long extraction batch | high | mitigate | Sub-batch to the SIDECAR's own `MAX_TEXTS=256` and keep each text under `MAX_TEXT_CHARS` via `chunk_chars=360`; call the SAME validated `/extract` endpoint with the SAME bounds respected. Do not raise the bounds and do not add an unbounded endpoint (research Security Domain, V5). |
