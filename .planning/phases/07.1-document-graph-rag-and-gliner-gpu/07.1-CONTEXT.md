## Phase Boundary

ROADMAP.md carries this phase as `Goal: [Urgent work - to be planned]` / `Requirements: TBD`.
This discussion sets that scope. **Downstream agents must treat this section as the phase
goal until ROADMAP.md is updated (see D-18).**

**Phase 07.1 delivers:**

1. **The unconditional wins** — correct entity resolution (both memory and document paths),
   GLiNER on GPU with a real batching path, and the whole-document→HTTP 400 ingest fix.
   These stand on their own merit and are *not* contingent on any graph bet.
2. **A decision artifact** — a MuSiQue-style multi-hop eval with a **pre-committed** GO/NO-GO
   threshold, which decides whether a document graph channel ships at all.
3. **A flag-gated (default-OFF) graph channel** — real, tested, tenant-scoped code, built only
   to the grade needed for the eval to read it. Productionized only on a GO.

**This phase does NOT commit to shipping a document graph channel.** Three independent research
lines and the project's own spike 003 converge: the channel adds ~0 lift on grounded-passage
retrieval, and its value is an unproven-for-us distinct capability. The phase buys the
*decision*, not the feature.

**Explicitly out of scope:**
- Changing the embedding model (PROJECT.md Out of Scope — invalidates the ARC-01 baseline and
  the ARC-09 GO gate that authorized the TuringDB cut).
- The `gliner_multi-v2.1` multilingual swap — spike 003 measured Italian NER as adequate
  (9.8 entities/chunk of real Italian domain terms); no evidence of need, and it is a sidecar
  rewrite (v2.1/transformers.js ONNX ≠ fast_gliner/gliner2), not a model swap.
- The broader TEST-08 extraction failure-mode suite (timeouts, malformed responses, rate
  limiting) — stays mapped to Phase 9.
- Rewriting retrieval-fusion scoring or ranking weights (PROJECT.md: "stabilize, don't
  redesign scoring"). The document_search fusion port (D-14) is a *shape* port with flag-OFF
  behavior preservation, not a scoring redesign.

<decisions>
## Implementation Decisions

### Phase Outcome & Gate

- **D-01:** **Eval-first, build-gated.** 07.1 delivers the unconditional wins (ER fix, GLiNER
  GPU, HTTP 400 fix) plus the multi-hop eval with a pre-committed threshold. The graph channel
  is built flag-gated and default-OFF; it is productionized only if the gate reads GO.
  *Rationale:* spike 003's own recommendation ("gate any build on a NEW multi-hop eval, not
  `real_document_benchmark`") and all three research lines independently agree.

- **D-06:** **The gate measures EFFICACY; demand is recorded as UNMEASURED.** A MuSiQue-style
  eval is multi-hop *by construction*, so it proves the channel works — never that anyone will
  ask such questions. The gate artifact MUST state this explicitly and cite spike 003's 0/15 as
  the only real-traffic signal we have (which points at single-passage lookups). A GO is
  therefore an efficacy result **plus a consciously-accepted demand bet**, and must be written
  down as such. Do not let the eval's construction launder into a demand claim.

- **D-15:** **Pre-committed threshold — three arms, GO requires ALL four conditions.**

  Arms (all reranked by the real BGE reranker over the same pool, so the only difference is the
  channel — mirroring spike 003's isolation design):
  - `fused_base` — dense + lexical → rerank (today's shape, via fusion)
  - `entity_boost` — mem0-style **single-hop** entity inverted-index boost, hub-damped (D-03)
  - `fused_graph` — multi-hop entity-bridge graph channel

  **GO requires all of:**
  1. `fused_graph` beats **the best of the two cheap arms** by **≥ +0.10 mean recall@10** of the
     bridging passage set (not merely beating `fused_base` — see rationale).
  2. Improves **≥ ⅓ of questions** with **zero regressions**.
  3. **No regression > 0.03** on spike 003's existing 15-question frozen yardstick (matching
     Phase 6's locked epsilon).
  4. **Latency ≤ +20%** vs the best cheap arm (spike 003 measured +10% for zero gain).

  *Rationale for the `entity_boost` control arm:* mem0's replacement for its **deleted** graph
  memory — a name-keyed entity inverted index with a hub-damped single-hop boost — apparently
  *beat the graph it replaced*. Multi-hop must beat that cheap baseline, not just beat
  no-graph, or we ship complexity a simpler mechanism already delivers. The arm is nearly free
  once the substrate (D-12) exists.

  *Rationale for a strict bar:* the eval is stacked **in the channel's favour** (multi-hop by
  construction, each question verified unanswerable from one chunk). If it cannot win here it
  cannot win anywhere. Meanwhile the literature ceiling is small — HippoRAG 2, the leading PPR
  system **with an LLM write path**, gets MuSiQue +2.9 F1 and **HotpotQA +0.2**; our substrate
  (NER, no triples) is weaker. The named failure mode is: build it, measure +0.2, call it a win
  because it's already built. Conditions 1 and 2 exist to prevent exactly that.

- **D-07:** **Eval authorship — Claude composes, user verifies.** Claude mines the real corpus
  for cross-document entity bridges, composes candidate two-hop chains per MuSiQue's recipe
  (hop-2's *question* depends on hop-1's *answer*), and **mechanically verifies each is
  unanswerable from any single chunk** by running it against the existing dense+lexical+rerank
  baseline — if the current stack answers it from one passage, it is not multi-hop and is cut.
  That verification is the whole value and is the property spike 003's 15 questions lack. The
  user reviews survivors for whether a real user would plausibly ask them — the judgment Claude
  cannot make.

- **D-08:** **Metric — recall@k of the bridging passage set.** Deterministic, cheap, isolates
  retrieval from generation, matches the repo's existing gate discipline (`e2e_score.py`,
  `gate_diff.py`). **No LLM-as-judge**: the research recommends against it for multi-hop
  *retrieval*, and judge bias is specifically cited (arXiv 2506.06331) as a reason published
  GraphRAG gains are overstated.

- **D-16:** **Gate-fail disposition — delete the channel, keep the substrate, write the
  verdict.** On NO-GO: delete the flag-gated channel and the fusion port; **keep** the substrate
  (the ER fix path needs it; `entity_boost` may use it) and **keep** the eval as a permanent
  artifact. Write a FINDINGS verdict citing arXiv 2502.11371 + 2506.05690 + spike 003's 0/15 +
  our own measured number, so the question is not relitigated every quarter (the research
  explicitly recommends recording the closure). Dormant channel code is rejected: it would tax
  invariants #1/#4/#5/#6 forever against an upside our evidence says does not exist.

### Entity Resolution (the crux)

- **D-02:** **Name-only entity key + `type_observations` map.** Key on
  `stable_id("ent", user_identifier, canonical_name)`. `entity_type` becomes a **non-identity
  attribute**, plus a `type_observations` map on the vertex (e.g. `{product: 7, library: 3,
  framework: 1}`) kept as a rerank/filter signal **and** as the evidence base for a future
  deliberate homograph split.
  *Rationale:* current `(entity_type, canonical_name)` keying at `temporal_graph.py:137` splits
  one entity into many under GLiNER type-drift (`arcadedb` → product/library/framework),
  measured at **3× fewer cross-doc bridges** (spike 001). mem0 keys on normalized name in
  **both** its LLM graph (`(name, user_id)`, single `__Entity__` label) and its NER replacement;
  the industrial literature calls name-only "the floor — do this regardless."
  *Accepted cost:* true homographs merge (`Paris`/person + `Paris`/city). Accepted because our
  corpus is Italian legal/technical, where measured fragmentation is the real problem and
  homographs are rare; `type_observations` preserves the evidence to split later if needed.
  *Rejected alternative:* neo4j's POLE+O coarse-type collapse (closed 5-type vocab with an
  `("OBJECT", label)` fallback, drift → non-identity `subtype`). It preserves homograph
  splitting, but needs a label map to maintain and **drift across coarse buckets**
  (product→OBJECT vs organization→ORGANIZATION) would still re-fragment. Recorded as the
  fallback if name-only proves too coarse.

- **D-03:** **Hub penalty — mem0's quadratic damping, mandatory.** Apply
  `1/(1 + 0.001 * (n-1)^2)` (mem0 `main.py:1754`) to the graph channel's per-candidate score
  **before** RRF, where `n` = linked chunk count. An entity in 5 chunks keeps ~0.98 weight; one
  in 100 drops to ~0.09.
  *Rationale:* both the mem0 and industrial research lines independently flagged this as
  non-optional. **Merging harder creates hubs**: collapsing `arcadedb`'s three fragments makes
  one node linked to 3× more chunks, and bridging value is inversely proportional to entity
  ubiquity. The ER fix **regresses precision without this counterweight** — it cannot ship
  without it.

- **D-04:** **Fix BOTH paths; rebuild projections.** One change at `temporal_graph.py:137` fixes
  the existing memory graph **and** gives document chunks a correct substrate. Entity, fact, and
  community projections rebuild via the existing `memory_rebuild_vector_projection` +
  `memory_rebuild_communities` tools on Phase 4's D-07 versioned atomic-swap foundation.
  *Rationale:* CLAUDE.md's fix-on-touch rule ("NOT MY WORK is not an excuse"); spike 001 flagged
  the memory-graph fragmentation explicitly as a fix-on-touch candidate. Leaving it would create
  two divergent entity-keying schemes in one codebase.
  *Risk:* touches the E2E-gated memory search path — guarded by D-17.

- **D-05:** **SAME_AS synonym edges — decide after the ER re-measure.** Not a yes/no now.
  Land the name-only key, **re-run spike 001's bridge measurement**, and let the residual
  fragmentation *shape* decide. Only if what remains is alias-shaped (`ArcadeDB` vs `Arcade DB`
  vs `arcade-db` — which `canonicalize_entity_name` may already cover) reach for HippoRAG 2's
  embedding-threshold `SAME_AS` (no LLM; we already have the embedder + native `LSM_VECTOR`).
  If built, start at a **tight** threshold: mem0's *surviving* value is **0.95**; its deleted
  graph used 0.7. This is a recorded decision point the plan must schedule, not skip.

### GLiNER GPU + Chunking

- **D-09:** **CUDA image + binding proof + real batching.** Deliverable is: a CUDA-capable
  Dockerfile (`onnxruntime-gpu`), **a test that FAILS if the CUDA execution provider is not
  actually bound**, and a real batched inference path investigated against `fast_gliner`'s API.
  Same model — **no swap** (spike 003: Italian NER adequate). Measure before/after on spike
  003's 167-chunk set.
  *Two load-bearing findings from the scout:*
  - `gliner_provider.py:45` **already defaults `GLINER_DEVICE` to `cuda`**, validates
    `cpu|cuda` (`:54-55`), and passes `execution_provider=device` into
    `FastGLiNER2.from_pretrained` (`:89`). Compose explicitly overrides it to `cpu`. But
    `docker/gliner-provider.Dockerfile` is `python:3.12-slim` with **no CUDA runtime** — so
    setting `cuda` today would hit ONNX Runtime's **silent fallback to CPUExecutionProvider**.
    You would "enable GPU", measure nothing, and get no error. The binding-proof test exists
    precisely to make that impossible (repo's no-skip-as-green discipline).
  - `gliner_provider_extraction.py:61`: `del batch_size  # FastGLiNER2 currently accepts one
    text per inference.` — **`GLINER_BATCH_SIZE` is inert.** Compose sets `1`, code defaults to
    `8`; **both are decorative**. There is no batching on any device. This is a major reason
    spike 003 saw 2215 chunks take >27 min. GPU without batching leaves most of the throughput
    on the table. Verify whether that comment is still true of `fast_gliner==0.2.1` before
    designing the path.

- **D-10:** **Fix the whole-doc → HTTP 400 gap in 07.1.** `ingest_document_text` →
  `_process_text_for_storage` runs GLiNER `process_many` on the **whole document text**; a
  300-page legal PDF as one payload returns HTTP 400 from the sidecar (spike 003 crashed on
  this; the Phase-3 baseline dodged it by running GLiNER off).
  *Rationale:* per-chunk extraction is exactly what the (:Chunk)-[:MENTIONS]->(:Entity)
  substrate needs anyway — the fix is **on the critical path**, not adjacent to it. CLAUDE.md's
  fix-on-touch rule agrees. The broader TEST-08 failure-mode suite stays in Phase 9 where it is
  mapped.

- **D-11:** **GPU contention — measure first, then decide.** `agentmemory-embed` and
  `agentmemory-rerank` already take `gpus: all` + `devices: driver: nvidia, count: all` +
  `--device CUDA0 --gpu-layers all`. Adding GLiNER makes three models contend for one GPU (the
  Phase 6 capture ran on an RTX A2000). Enable GLiNER on GPU and **measure** the effect on
  embed/rerank latency and the E2E score gate **before** tuning anything.
  *Rationale:* the repo's discipline is measure-don't-guess, and we have a documented latency
  baseline (3345 ms mean vs the 5.4 s bar) with headroom to detect a regression.
  *Note:* CI must still degrade GPU tiers to a visible compile/stub floor on GPU-less runners
  (DOCK-06, PROJECT.md constraint) — never silent green.

### Graph Channel Design

- **D-12:** **Production substrate + flag-gated channel (default OFF).** Chunk entity extraction
  and `(:Chunk)-[:MENTIONS]->(:Entity)` edges ship as **production** — they are needed by the ER
  fix path anyway and are cheap (spike 001: reuse `EntityProcessor` + `stable_id` +
  `_write_many`). The graph **channel** ships behind a default-OFF flag: real code, real tests,
  tenant-scoped, but inert unless enabled. The eval flips the flag to measure.
  *Note the wrinkle this resolves:* "eval-first" cannot mean "no channel code" — the gate needs
  a measurable channel. It means deciding what *grade* of code the gate reads.

- **D-13:** **RRF via the existing fusion layer.** Feed the graph channel into
  `retrieval_fusion.fuse_rankings` — the same weighted RRF `memory_search` uses. The
  `"graph": 0.5` weight already exists in `store_core.py`'s `fusion_weights`, and hop decay is
  already proven at `store_evidence.py:438` (`raw_score = seed_score * confidence / hop`).
  *Rationale:* CLAUDE.md's FOLLOW EXISTING PATTERNS rule; mem0's lesson that the entity channel
  should be **sub-full weight** (`ENTITY_BOOST_WEIGHT = 0.5`) corroborates the existing 0.5.
  *Known-better alternative, deliberately not taken:* HippoRAG 2 does **not** RRF a graph
  channel — it uses Personalized PageRank with dense similarity as reset mass and soft
  continuous damping, warning that fixed hop-caps over a neighbor list are "the classic
  neighbor-flood". Rejected for 07.1 because it would invent a new ranking mechanism in a
  codebase whose entire retrieval story is RRF, to chase a lift unproven for us. **Recorded as
  the upgrade path if the eval shows the channel matters and RRF is the limiter.**

- **D-14:** **Port `document_search` onto the fusion shape, gated behind the flag.**
  `search_documents` (`store_documents.py:448-585`) is `blend_hybrid_score` →
  `_rerank_documents` — a *different architecture* from `memory_search`'s
  `RetrievalCandidate` → `fuse_rankings` → `diversify_fused` → rerank
  (`store_search.py:279-314`). A graph channel needs the fusion shape.
  **Hard constraint:** with the flag **OFF**, ranking behavior must be **preserved exactly** —
  the E2E score gate and Phase 6's locked metrics staying green is the proof the port is
  behavior-preserving.
  **Must fix explicitly:** `diversify_fused(max_per_source=1)` is correct for memories (one hit
  per memory) but would **cap each document at a single chunk**. Documents need
  `max_per_source > 1`. This is a real trap, not a detail.

- **D-17:** **Mini migration-correctness gate for the rebuild.** Apply Phase 6's own discipline
  to the ER change: after rebuild, the E2E score gate must stay **19/19 at 10.0**, and re-run
  the frozen-question benchmark to confirm the document path is untouched (flag OFF). Record it
  as a gate artifact in the shape of Phase 6's `GATE.md`. The machinery already exists
  (`gate_diff.py`, `e2e_score.py`).
  *Precision note:* the ER fix affects **memory** search (which uses entity/graph/community
  channels). `document_search` has **no entity channel today**, so the frozen-question document
  benchmark cannot move from the ER fix alone — it becomes relevant only once the channel flag
  turns on. The E2E gate's 19 checks are the guard for the memory-side change.

### Research sources (2026 industrial practice — cited in D-01/D-06/D-15/D-16)
- https://arxiv.org/html/2502.11371v3 — *RAG vs. GraphRAG: A Systematic Evaluation*. Controlled
  settings: RAG **beats** graph on single-hop factoid (NQ 64.78 vs 60.04 F1); graph wins
  multi-hop narrowly (HotpotQA 61.66 vs 60.04). Complementarity, not dominance.
- https://arxiv.org/html/2506.05690v3 — *When to Use Graphs in RAG*. Graph "introduced redundant
  information, actually degrading performance" on simple fact retrieval; 40,000+ tokens/query
  vs 879 vanilla.
- https://arxiv.org/html/2502.14802v1 — *HippoRAG 2*. The no-LLM **synonym-edge ER** pattern
  (D-05) and PPR-with-dense-reset-mass (D-13's upgrade path). Gains: MuSiQue +2.9 F1,
  **HotpotQA +0.2**.
- https://www.microsoft.com/en-us/research/blog/lazygraphrag-setting-a-new-standard-for-quality-and-cost/
  — **validates our NER-only substrate**: noun-phrase extraction at index time, defer LLM to
  query time; indexing at 0.1% of full GraphRAG cost.
- https://arxiv.org/abs/2506.06331 — *An Unbiased Evaluation Framework for GraphRAG*. Published
  gains "much more moderate than reported previously"; LLM-judge bias named (grounds D-08).
- https://arxiv.org/pdf/2510.14271 — *Less is More: Denoising KGs for RAG*. Entity fragmentation
  "leads to disconnected subgraphs that harm retrieval"; over-dense graphs inject noise.
- https://www.alphaxiv.org/overview/2108.00573v3 — *MuSiQue*. The question-construction recipe
  D-07 copies; the DiRe cheatability metric.

**Not verified — do NOT plan against:** the "GraphRAG 86% vs vector 32% on multi-hop" /
"vector accuracy → 0%" figures widely quoted online trace to vendor blogs with no reproducible
methodology and contradict the controlled evaluations above by an order of magnitude.

<deferred>
## Deferred Ideas

- **`embedl/Cosmos-Reason2-2B-W4A16-Edge2-FlashHead` as a one-model-for-all replacement** —
  **not viable.** It is a quantized **robotics vision-language model** (derivative of
  `nvidia/Cosmos-Reason2-2B`, itself on `Qwen3-VL-2B-Instruct`), Text+Image/Video→Text, trained
  for physical-AI/embodied reasoning. It emits **no embeddings** (fatal — `FlashHead` literally
  replaces the dense LM head for generation throughput); rerank/NER via prompting would be far
  slower than BGE/GLiNER. License stacks NVIDIA Open Model License + **Embedl Models Community
  Licence v1.0 (no redistribution)** — contradicts a repo that chose ArcadeDB on licensing and
  has public-launch intent. Domain mismatch: video/spatial reasoning vs Italian legal/technical
  text.
- **`GnLOLot/MiniCPM5-1B-Claude-Opus-Fable5-Thinking-GGUF`** — **not viable for this purpose.**
  Base `openbmb/MiniCPM5-1B` is legitimate (SOTA-for-1B, Apache-2.0, 128K, MCP + tool calling),
  but the linked fine-tune is a random-account upload "post-trained on Fable 5 data" with **no
  methodology, no provenance, no benchmarks**; the card analysis notes the "Claude-Opus" naming
  "appears promotional rather than descriptive of actual model lineage." Also a **generative
  LLM → no embeddings**. Training on frontier-model outputs carries live legal risk for a
  product with public-launch intent.
- **Qwen3-Embedding + Qwen3-Reranker consolidation** — the **viable** version of "fewer
  sidecars": one purpose-trained base family covering embed + rerank, license-clean. Note
  `Qwen3-Reranker-0.6B-q8_0.gguf` is **already pinned** in `compose.yaml`'s model-init, and an
  `embeddinggemma-eval` profile already exists — the A/B machinery is partly there. **Blocked
  this milestone:** PROJECT.md marks embedder changes Out of Scope (invalidates the ARC-01
  baseline + ARC-09 GO gate). Natural home: Phase 9's "A/B embedding-model swap / canary /
  rollback using the D-07 foundation".
- **Local small LLM for LazyGraphRAG-style query-time reasoning** — the research says our
  NER-at-index substrate is the frontier and the LLM belongs at *query* time. A genuine future
  option, but a separate phase and orthogonal to 07.1's gate.
- **PPR with dense-derived reset mass** (HippoRAG 2) — the literature's better ranking design;
  recorded in D-13 as the upgrade path if the eval shows the channel matters and RRF is the
  limiter.
- **neo4j's POLE+O coarse-type collapse** — recorded in D-02 as the fallback if name-only
  proves too coarse for our corpus.
- **`SAME_AS` + tombstone merge** (`graph/queries.py:1001-1032`) — the only post-hoc merge shape
  compatible with deterministic IDs (re-point edges, set `merged_into`, **never delete the
  node**; readers filter tombstones). Only if D-05's re-measure demonstrates a post-hoc
  correction need. If ported: add the `Chunk` label to the transfer leg (neo4j's only matches
  `(m:Message)`), run inside one `run_in_transaction` (invariant #5), and carry
  `user_identifier` on every leg (invariant #1).
- **`gliner_multi-v2.1` multilingual swap** — explicitly **not warranted** (spike 003: Italian
  NER adequate at 9.8 entities/chunk of real domain terms; the only noise was Wikipedia HTML
  chrome, a conversion artifact already fixed by quick task `260716-lel`). Would be a sidecar
  rewrite, not a swap. Reopen only on evidence of an actual Italian NER gap.
- **`vector.fuse` native RRF** (available on 26.7.1, spike 002) — a real lever for server-side
  dense+sparse fusion; Phase 4 deferred it by choice. Does not do the graph channel. Not
  needed for 07.1's gate.
