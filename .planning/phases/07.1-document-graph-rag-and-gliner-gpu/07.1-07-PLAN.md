<objective>
Build the deterministic scoring primitives the multi-hop eval needs: set-recall over a bridging
passage set (D-08's metric), the single-passage-answerability rejection filter (D-07's mechanical
cut), and a loader for the new multi-hop question schema.

Purpose: this is the only genuinely new code in GRAPH-03, and it is pure — no providers, no GPU, no
corpus. Building it first (wave 1) means the eval driver in plan 15 assembles tested parts under
real GPU/corpus conditions instead of debugging arithmetic there. D-08 chose recall@k of the
bridging passage set precisely because it is deterministic, cheap, isolates retrieval from
generation, and matches the repo's existing gate discipline — and explicitly **rejects
LLM-as-judge**, which arXiv 2506.06331 names as a reason published GraphRAG gains are overstated.
Output: `scripts/multihop_eval_scoring.py` + its test file.

<feature>
  <name>Bridging-set recall@k, the rejection filter, and the multi-hop question schema</name>
  <files>scripts/multihop_eval_scoring.py, tests/test_multihop_eval_scoring.py</files>
  <behavior>
    `bridging_set_recall_at_k(hits, bridging_passages, k)` -> float in [0.0, 1.0]:
    - 3 required bridging passages, all 3 contained in the top-k hits -> 1.0
    - 3 required, 2 found -> 2/3 (assert to full float precision, not rounded)
    - 3 required, 0 found -> 0.0
    - A passage found only at rank k+1 does NOT count (the k cut is exclusive of later hits)
    - Duplicate passages in `bridging_passages` are de-duplicated before the fraction is computed (N is the count of DISTINCT required passages), so a repeated passage cannot inflate recall
    - One hit containing TWO required passages satisfies both (credit is per-passage, not per-hit)
    - Empty `bridging_passages` raises ValueError — a multi-hop question with no bridging set is a data error, not a 1.0
    - Matching reuses `real_document_benchmark_scoring.normalize_text`'s normalization, so whitespace/case/punctuation variation does not cause a false miss
    `single_passage_answerable(hits, answer, evidence_quote)` -> bool:
    - True when the FINAL answer (or its evidence quote) is contained in a SINGLE top-ranked hit — meaning today's dense+lexical+rerank stack already answers it from one passage, so the question is NOT multi-hop and must be cut (D-07)
    - False when the answer only emerges across multiple hits
    - This is a DIFFERENT function from the metric above and must not delegate to it (research Pitfall 5)
    `load_multihop_questions(path)`:
    - Accepts rows carrying `bridging_passages: list[str]` and rejects rows missing it, with an error naming the offending row
    - Rejects a row whose `bridging_passages` has fewer than 2 entries (a one-passage "multi-hop" question is a contradiction)
    - Does NOT accept or modify the existing Phase-6 single-`evidence_quote` schema
    `summarize_multihop_results(per_question_by_arm)`:
    - Returns per-arm mean set-recall@k across questions
    - Returns, per arm pair, the count of questions improved and the count regressed vs a named baseline arm (D-15 condition 2 needs both, and "zero regressions" needs the regression count specifically)
  </behavior>
  <implementation>
    A new pure-Python module importing the reusable primitives from
    `real_document_benchmark_scoring.py`. No new algorithm — set-recall is the natural adaptation of
    the existing `evidence_rank` containment pattern.
  </implementation>
</feature>
