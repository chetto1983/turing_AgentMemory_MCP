<objective>
Build the durable pseudonymous registry that distinguishes new/resumable provisioning from
previously-ready tenant databases and binds the whole deployment to one immutable naming
version/key fingerprint (D-04 and D-15).

Purpose: HMAC routing is deliberately catalog-free, but restart safety still needs durable
evidence that a missing previously-ready database represents data loss, not a new tenant.
Output: A transactional SQLite registry with deterministic failure behavior and TDD coverage.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Process routing state to local SQLite registry | Lifecycle evidence crosses from transient process state into durable operator-controlled storage. |
| Registry reopen to future provisioning | Persisted metadata decides whether a database may be created, resumed, or treated as missing data. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-02-01 | Tampering | registry metadata/state | high | mitigate | Task 2 validates immutable schema/naming/key metadata and legal state transitions in explicit transactions; corruption/drift tests require fail-closed behavior. |
| T-05-02-02 | Information Disclosure | registry file and runtime status | high | mitigate | API/schema accept only opaque identities; raw-byte and status tests prove no exact tenant identifier is persisted or returned. |
| T-05-02-03 | Denial of Service | concurrent SQLite writers | medium | mitigate | Finite busy timeout, WAL, connection-per-operation, and short `BEGIN IMMEDIATE` transactions are exercised under concurrency. |
| T-05-02-04 | Repudiation / Data-loss camouflage | ready-to-missing lifecycle | high | mitigate | Durable `ready` state cannot be silently demoted or recreated; Plan 05-04 consumes this evidence to fail closed on a missing database. |
</threat_model>
