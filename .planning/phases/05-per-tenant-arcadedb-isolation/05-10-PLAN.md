<objective>
Close the third "missing" item of verifier gap 1: *"Add read/write/update/delete/document/background
adversarial binding tests."*

Plan 05-09 made the guard correct where it is called. This plan makes it **reachable and ordered**.
Two structural defects block the must-have:

1. **Seven public methods never call `_require_user` at all** (confirmed by source audit):
   `add_entity`, `add_preference`, `add_fact` (store_memory_write), `update_memory`, `delete_memory`
   (store_memory_read), `get_context`, `delete_document` (store_documents). The binding introduced in
   05-09 is simply never consulted on those paths — `add_*` reach `_write_memory`, which only calls
   `_ensure_user` (a database read/create, not a validation guard).
2. **Where the guard IS called, it runs inside the span**, e.g. `store_memory_write.py:54-57` opens
   `memory.store_message` and only then calls `_require_user`. The must-have requires rejection
   *before* any span or audit event, so a rejected foreign identifier currently still emits a span.

Purpose: a binding that is bypassable on 7 of 18 public paths is not defense in depth.
Output: guard-first ordering across the store surface, guards on the uncovered paths, an adversarial
matrix, and a static catalog test that fails if a future method skips the guard.
</objective>
