<objective>
Close verifier gap 1 (BLOCKER, ARC-07): `TenantRouter.resolve()` binds only the ArcadeDB client,
so a tenant-A store view accepts tenant B's *valid* `user_identifier` and executes B-scoped
operations inside A's physical database.

This plan makes CLAUDE.md invariant #3 mechanical: the bound client, the manifest, and the explicit
`user_identifier` must all agree. It introduces a keyed `TenantBinding`, threads it through the
router into every routed store, and converts `_StoreCore._require_user` from a static syntax check
into an instance-level binding check that fails closed.

Purpose: physical database separation currently has no logical counterpart. Without the binding, a
misrouted or hostile identifier silently reads and writes the wrong tenant's database.
Output: `tenant_binding.py`, a bound `_require_user`, router wiring, and adversarial regressions.

Scope note: this plan makes the guard *correct where it is called*. Plan 05-10 makes the guard
*reachable on every public path* (7 methods currently never call it) and enforces call ordering.
</objective>

- `TenantProvisioner.naming_key: bytes` — already held by the provisioner the router owns.
- `TenantRouter._validate_provisioned` — the existing precedent for fail-closed binding checks
  whose error message names only `database_name`.
- Exception style precedent: `class RequestFailure(ValueError)` — new errors subclass a builtin.

`_StoreCore.__init__` currently rejects `turing_home`/`runtime_signals` alongside
`shared_dependencies`. `tenant_binding` is per-tenant state (like `client`), NOT shared state, so it
is a separate keyword argument and MUST NOT be added to `StoreSharedDependencies` or to the
`shared_dependencies()` bundle — that bundle is reused across tenants and would poison the binding.
</interfaces>
