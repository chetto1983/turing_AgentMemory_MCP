<objective>
Close the one verified Phase 4 gap: retire the WRITE side of the legacy SQLite-FTS5 sparse-index outbox (`prepare`/`commit_batch`/`replay`/`discard_prepared`), which is still live in `store_memory_write.py`, `store_memory_read.py`, and `store_rebuild.py` whenever `AGENTMEMORY_FUSION_ENABLED=true`, even though 04-07 already fully retired the READ side and nothing in the codebase reads this projection anymore (04-VERIFICATION.md, Success Criterion #3 / ARC-06).

**Disposition (confirmed by direct source inspection, not assumed): retire-directly.** Both `store_search.py` and `store_evidence.py` (04-07) contain zero references to `sparse_index` -- `store_evidence.py`'s `_lexical_evidence`/`_merged_lexical_scores` feed the `bm25` RRF channel exclusively from ArcadeDB's native `vector.sparseNeighbors` (the `LSM_SPARSE_VECTOR` channel populated by `sparse_encoder.sparse_vector()`) and native `SEARCH_INDEX` (Lucene). `store_memory_write.py`/`store_memory_read.py`/`store_rebuild.py` already write `lexical_tokens`/`lexical_weights` unconditionally on every CREATE/UPDATE (not gated on `fusion_enabled` or `sparse_index`), so lexical retrieval already does not depend on the outbox in any way. There is no "ensure the native channel feeds fusion first" step needed -- it already does.

**A second, independent reason this is not merely cleanup:** `store_core.py`'s `bootstrap()` was changed in 04-04 to no longer call `sparse_index.initialize()` (confirmed by `tests/test_store_arcadedb_core.py::test_sparse_outbox_replay_calls_absent_from_source`, which asserts exactly this). On a fresh deployment volume, the SQLite outbox file therefore has no `sparse_meta`/`sparse_outbox` tables. `sparse_index.py`'s `_ready_connection()` raises `SparseSchemaMismatch` (a `RuntimeError` subclass, not caught by any of the four write call sites' `except (OSError, sqlite3.Error)` guards) the first time `.prepare()` is called on such a file. Concretely: `store_messages`'s `sparse_batch_id = self._prepare_sparse_projection(...)` call sits OUTSIDE its own try/except, so on a fresh volume with `AGENTMEMORY_FUSION_ENABLED=true` (a supported, dogfooded production configuration), every batch memory write would crash unhandled. Retiring these call sites is a correctness fix, not just dead-weight removal.

Purpose: Make Success Criterion #3 and REQUIREMENTS.md's ARC-06 row ("the SQLite-FTS5 outbox prepare/commit/replay path retired") literally true -- both already read correctly, only the code needs to catch up -- while proving lexical retrieval quality is unaffected because the native channels already carry it.
Output: `store_memory_write.py`/`store_memory_read.py`/`store_rebuild.py` free of outbox stage/commit/replay/discard calls; `store_rebuild_sparse.py` deleted; `store.py`'s mixin MRO updated; RED-then-GREEN regression tests proving the above; a new end-to-end test proving lexical retrieval is unaffected; docs/CHANGELOG updated for the retired write path.
</objective>
